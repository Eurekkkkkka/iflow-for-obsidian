# iFlow for Obsidian Windows 隐式启动改进方案

## 1. 背景与现象

在 Windows 环境打开 iflow-for-obsidian 插件后，会弹出一个标题为“node”的控制台窗口，窗口内容类似：

```text
[ACP] starting server with: {
  "workspace": "C:\\Program Files\\Obsidian",
  "port": "8090",
  "verbose": false,
  "disableTools": false,
  "stream": true
}
🚀 iFlow ACP Server running at ws://127.0.0.1:8090/acp
[WS 1] open connection (total=1) peer=undefined origin=app://obsidian.md
[WS 1] About to call startEmbeddedGeminiPeer
[ACP Web Peer] ACP adapter factory initialized
[WS 1] close event code=1005 reason= (remaining=0)
```

用户期望是：插件在后台隐式启动 iFlow ACP 服务，不弹出任何系统命令行窗口。

## 2. 现状结论

### 2.1 插件确实通过 ACP 方式调用后台服务

现有代码中，Obsidian 端并不是直接通过 stdin/stdout 和 CLI 交互，而是通过 WebSocket 连接 ACP 服务：

- 启动逻辑位于 [src/main.ts](src/main.ts#L152)
- ACP 连接逻辑位于 [src/infra/acp/acpClient.ts](src/infra/acp/acpClient.ts#L25) 和 [src/infra/acp/acpClient.ts](src/infra/acp/acpClient.ts#L51)

因此，你看到的控制台窗口本质上是“本地 ACP 服务进程”的控制台，而不是 Obsidian 主进程本身。

### 2.2 当前代码虽然做了隐藏处理，但还不够稳妥

当前 [src/main.ts](src/main.ts#L152) 已经使用了下面的思路：

1. `spawn('iflow', [...])`
2. `windowsHide: true`
3. `stdio: 'ignore'`
4. `detached: true`
5. `unref()`

这套做法对“真正的 exe”通常足够，但对 Windows 上的 npm CLI 并不一定成立。

## 3. 根因分析

### 3.1 Windows 上的 iflow 并不一定是原生 exe

在这台机器上，系统命令解析结果表明：

1. `iflow` 优先解析到 PowerShell 外壳脚本 `iflow.ps1`
2. 同时存在 `iflow.cmd` npm shim
3. 两者最终都会转发到 Node 入口脚本

换句话说，当前启动链路不是：

`Obsidian -> iflow.exe`

而更接近：

`Obsidian -> iflow.ps1 / iflow.cmd -> node.exe -> entry.js --experimental-acp`

### 3.2 为什么会看到“node”窗口

根因不是 ACP 本身，而是 Windows 的 CLI 启动层级：

1. `spawn('iflow', ...)` 启动的可能只是 shim 脚本，不是最终进程。
2. `windowsHide: true` 只能保证“当前被 spawn 的那一层”尽量不展示窗口，无法绝对约束后续 shim 再拉起的 `node.exe`。
3. 当 npm shim 或 PowerShell 脚本转发到 `node.exe` 时，真正获得控制台的可能是 `node.exe`，所以用户看到的窗口标题是“node”。
4. 这也是为什么日志内容明明是 ACP Server，但窗口标题却显示 node。

### 3.3 结论

**当前实现只能做到“尽量隐藏”，不能做到“必定不弹窗”。**

如果目标是“必定不会影响其他功能，同时在 Windows 上稳定隐式启动”，就不能继续依赖 `iflow` 这个抽象命令名，而必须解析到最终的真实启动目标。

## 4. 必定不影响其他功能的改进方案

## 4.1 设计原则

本次修复必须满足：

1. 只影响 Windows 启动分支
2. 不改变 ACP 协议、不改变端口、不改变 WebSocket 路径
3. 不改变 Unix/macOS/Linux 的现有行为
4. 不改变 `ensureSdkRunning()`、连接检查、自动启动、重启命令等上层逻辑
5. 只替换“Windows 进程拉起方式”这一处实现细节

## 4.2 推荐方案：绕过 shim，直接启动 node + CLI entry.js

### 核心思路

在 Windows 平台，不再执行：

```text
spawn('iflow', ['--experimental-acp', '--port', '8090', '--stream'])
```

而是改成：

1. 先解析出 iflow 的真实安装根目录
2. 直接找到 CLI 的 `entry.js`
3. 直接找到要执行的 `node.exe`
4. 用 `spawn(nodeExecutable, [entryJs, '--experimental-acp', '--port', port, '--stream'], { shell: false, windowsHide: true, detached: true, stdio: 'ignore' })`

### 为什么这个方案更稳

因为这样启动链路会变成：

`Obsidian -> node.exe -> entry.js --experimental-acp`

中间不再经过 `.ps1` 或 `.cmd` shim，Windows 窗口控制权就完全落在当前 `spawn` 的那一个真正进程上，`windowsHide: true` 才能稳定生效。

## 4.3 推荐模块拆分

建议新增一个只负责 Windows 启动目标解析的模块，例如：

- `src/app/windowsIflowLauncher.ts`

其职责只做两件事：

1. 解析 Windows 下的 iflow 实际可执行目标
2. 生成最终 `command + args + spawnOptions`

建议接口：

```ts
export interface ResolvedWindowsIflowLaunch {
  command: string;
  args: string[];
}

export function resolveWindowsIflowLaunch(port: number): ResolvedWindowsIflowLaunch
```

## 4.4 解析策略

建议按以下顺序解析：

### 方案 A：从 npm 全局 shim 反推

1. 定位 npm 全局 bin 目录中的 `iflow.cmd` 或 `iflow.ps1`
2. 取其所在目录作为 `basedir`
3. 推导出：
   - `node.exe`：`{basedir}/node.exe`，若不存在则回退到 `node`
   - `entry.js`：`{basedir}/node_modules/@iflow-ai/iflow-cli/bundle/entry.js`

### 方案 B：若找不到 sibling node.exe，则回退到系统 node

即：

```text
command = 'node'
args = [entry.js, '--experimental-acp', '--port', String(port), '--stream']
```

### 方案 C：解析失败时不做危险兜底

如果 `entry.js` 不存在，就不要再退回到 `spawn('iflow', ...)` 这种不稳定路径，而是：

1. 直接返回失败
2. 用 Notice 提示“检测到 Windows CLI shim，无法安全隐式启动，请检查 iflow CLI 安装完整性”

这样比“继续尝试但弹黑窗”更可控，也更符合“必定不影响其他功能”的要求。

## 5. 代码改造范围

## 5.1 只改启动层，不改协议层

无需改动：

- [src/infra/acp/acpClient.ts](src/infra/acp/acpClient.ts)
- [src/iflowService.ts](src/iflowService.ts)
- [src/chatView.ts](src/chatView.ts)

只需改动：

- [src/main.ts](src/main.ts#L152)
- 新增 `src/app/windowsIflowLauncher.ts`
- 可选保留或调整 `src/app/sdkSpawnOptions.ts`

## 5.2 startSdk 的目标形态

建议改成如下结构：

```ts
if (process.platform === 'win32') {
  const launch = resolveWindowsIflowLaunch(port);
  this.sdkProcess = spawn(launch.command, launch.args, {
    shell: false,
    detached: true,
    windowsHide: true,
    stdio: 'ignore',
  });
} else {
  this.sdkProcess = spawn('iflow', [
    '--experimental-acp',
    '--port', String(port),
    '--stream',
  ], {
    shell: false,
    detached: true,
    stdio: 'ignore',
  });
}
this.sdkProcess.unref();
```

关键点：

1. Windows 分支不再使用 `iflow`
2. Windows 分支不再依赖 shell
3. 直接启动真正的 `node.exe`
4. 仍保留 `unref()` + `stdio: 'ignore'`

## 8. 风险分析

### 低风险部分

1. ACP 协议链路本身无需修改
2. UI 层无需修改
3. 会话与聊天功能无需修改

### 真正的风险点

只有一个：**Windows 上 iflow CLI 的安装位置解析**。

因此最稳妥的做法不是“猜”，而是：

1. 明确地解析 npm shim 所在目录
2. 明确地验证 entry.js 是否存在
3. 解析失败时直接失败并给出可诊断提示

## 9. 实施建议

建议按以下顺序实施：

1. 先新增 `windowsIflowLauncher` 纯函数模块和单元测试
2. 再把 [src/main.ts](src/main.ts#L152) 中的 Windows 启动逻辑替换为“直接 node + entry.js”
3. 最后在真实 Windows 环境手测：
   - 冷启动插件
   - 设置页点击“重启 SDK”
   - Obsidian 重启后自动启动

## 10. 最终结论

是的，你看到的窗口本质上是 **ACP 后台服务对应的 Node 进程窗口**；插件业务层确实走的是 ACP WebSocket，而不是前台 CLI 交互。

但是，当前实现仍然依赖 Windows 的 npm shim，所以只能“尽量隐藏”，不能保证“绝对不弹窗”。

**真正稳妥、且不影响其他功能的方案是：Windows 下绕过 `iflow` shim，直接以隐藏模式启动 `node.exe + @iflow-ai/iflow-cli/bundle/entry.js`。**

这会把改动严格限制在“进程启动层”，不会影响 ACP 协议、聊天能力、工具调用、会话管理等现有功能。
