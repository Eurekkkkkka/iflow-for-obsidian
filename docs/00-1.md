# 顶部勋章配置控件工程化开发规范

## 1. 文档目标

本文档用于定义顶部勋章弹层中三个配置控件的工程化修复方案，使其从“前端可切换但用户难以感知是否真实生效”的半功能状态，提升为“会话内真实生效、跨会话可恢复、失败可感知、测试可覆盖”的完整功能。

本次治理对象仅限以下三个控件：

1. 模型切换
2. 模式切换
3. 思考开关

---

## 2. 问题定义

### 2.1 当前现象

当前顶部勋章弹层中的模型、模式、思考开关具备以下行为：

1. 用户点击后，UI 文案和摘要会变化。
2. 用户发送消息时，前端会尝试把当前值通过 RPC 下发给 iFlow 服务。
3. 控件状态会写入插件级别的 `lastUsedModel`、`lastUsedMode`、`lastUsedThinking`。

但同时存在以下关键缺陷：

1. 控件状态没有同步写入当前会话对象。
2. 切换会话时不会恢复该会话自己的模型、模式、思考状态。
3. 设置 RPC 失败时只有控制台日志，用户没有反馈。
4. 模型列表和模式列表是前端预置值，不保证与后端真实能力一致。

### 2.2 用户视角问题

从用户角度看，以上缺陷会被感知为：

1. 控件像“只有 UI，没有功能”。
2. 切到别的会话后配置混乱，不知道当前请求到底使用了什么配置。
3. 即使后端拒绝设置，界面也没有提示，用户只能靠猜。

---

## 3. 开发目标

本次开发完成后，应满足以下目标：

1. 用户切换模型、模式、思考开关后，当前会话配置立即更新。
2. 用户切换到任意已有会话时，顶部配置控件恢复为该会话保存的值。
3. 用户发送消息时，服务层使用当前会话对应的模型、模式、思考参数。
4. 若后端不接受设置类 RPC，用户能看到明确提示。
5. 核心行为具备自动化测试护栏。

---

## 5. 现有架构与责任边界

### 5.1 现有调用链

当前功能链路如下：

1. `chatView.ts` 负责模型、模式、思考控件的交互与当前视图状态。
2. `chatView.ts` 在发送消息时，将 `model`、`mode`、`thinkingEnabled` 传给 `iflowService.ts`。
3. `iflowService.ts` 在发送 prompt 前调用：
   - `session/set_model`
   - `session/set_mode`
   - `session/set_think`
4. `conversationService.ts` / `conversationStore.ts` 已具备 `model`、`mode`、`think` 的会话数据结构与更新接口。

### 5.2 责任划分

#### chatView.ts

负责：

1. 控件点击后的即时 UI 更新。
2. 当前会话配置同步。
3. 会话切换时恢复视图状态。
4. 设置失败时用户提示。

不负责：

1. 存储底层实现。
2. ACP 协议封装。
3. 后端能力发现。

#### conversationService.ts

负责：

1. 提供 `updateConversationSettings()` 会话配置更新接口。
2. 提供当前会话读取能力。

不负责：

1. 顶部 UI 刷新。
2. RPC 下发。

#### iflowService.ts

负责：

1. 在发送 prompt 前调用设置类 RPC。
2. 将后端错误暴露为调用方可感知结果。

不负责：

1. 决定当前会话该使用什么配置。
2. 维护会话切换 UI 状态。

## 6. 状态模型设计

### 6.1 配置来源优先级

发送消息时，配置来源优先级应为：

1. 当前会话的 `model` / `mode` / `think`
2. 当前视图内存状态 `currentModel` / `currentMode` / `thinkingEnabled`
3. 插件默认值 `lastUsedModel` / `lastUsedMode` / `lastUsedThinking`

工程实现上，为避免多源冲突，应采用以下约束：

1. 视图初始化时允许从插件默认值加载。
2. 一旦存在当前会话，视图状态以当前会话为准。
3. 用户点击控件时，必须同时更新视图状态和当前会话状态。
4. 插件默认值只作为“新开视图或无会话时”的兜底值。

### 6.2 会话状态字段

当前 `Conversation` 已具备：

1. `model`
2. `mode`
3. `think`

因此本次不新增 schema，不做数据迁移。

## 7. 详细功能规则

### 7.1 模型切换规则

当用户在模型下拉中选择某个模型时：

1. 更新 `currentModel`。
2. 更新模型按钮文案。
3. 更新顶部配置摘要。
4. 若存在当前会话，则调用 `updateConversationSettings(currentConversationId, { model })`。
5. 同步写入 `plugin.settings.lastUsedModel`。
6. 关闭下拉面板。

### 7.2 模式切换规则

当用户在模式下拉中选择某个模式时：

1. 更新 `currentMode`。
2. 更新模式按钮图标与文案。
3. 更新顶部配置摘要。
4. 若存在当前会话，则调用 `updateConversationSettings(currentConversationId, { mode })`。
5. 同步写入 `plugin.settings.lastUsedMode`。
6. 关闭下拉面板。

### 7.3 思考开关规则

当用户点击思考开关时：

1. 更新 `thinkingEnabled`。
2. 切换按钮 active 状态。
3. 更新顶部配置摘要。
4. 若存在当前会话，则调用 `updateConversationSettings(currentConversationId, { think })`。
5. 同步写入 `plugin.settings.lastUsedThinking`。

### 7.4 会话切换恢复规则

当用户切换到任意会话时：

1. 读取该会话的 `model`、`mode`、`think`。
2. 用会话值覆盖视图状态：
   - `currentModel`
   - `currentMode`
   - `thinkingEnabled`
3. 刷新顶部摘要。
4. 刷新模型按钮显示。
5. 刷新模式按钮显示。
6. 刷新思考按钮 active 状态。
7. 然后加载该会话消息列表。

### 7.5 新建会话继承规则

当用户创建新会话时：

1. 新会话默认继承当前视图状态中的 `model`、`mode`、`think`。
2. 新会话创建完成后，视图状态不应发生跳变。
3. 不允许出现“新会话继承全局默认值，但忽略当前已选配置”的行为。

### 7.6 发送链路规则

每次发送消息前：

1. `chatView.ts` 必须把当前视图状态传给 `iflowService.sendMessage()`。
2. `iflowService.ts` 继续使用现有顺序发送：
   - `session/set_mode`
   - `session/set_model`
   - `session/set_think`
   - `session/prompt`
3. 若其中任一设置失败，必须把失败结果向上返回或抛出，由 UI 层处理。
4. UI 层必须展示用户可见提示，不能只 `console.warn`。

### 7.7 失败提示规则

设置类 RPC 失败时推荐提示文案：

1. 模型失败：`模型设置未被 iFlow 接受，本次请求可能仍使用旧模型。`
2. 模式失败：`模式设置未被 iFlow 接受，本次请求可能仍使用旧模式。`
3. 思考失败：`思考设置未被 iFlow 接受，本次请求可能仍使用旧配置。`

提示方式：

1. 优先使用 `Notice`
2. 单次发送中多项失败时可合并提示，避免通知轰炸

## 8. 工程实现方案

### 8.1 最小实现路径

推荐最小改动方案如下：

1. 保留当前 UI 结构与服务调用方式。
2. 在三个控件的点击事件中加入“同步写入当前会话”的逻辑。
3. 在 `onConversationChange()` 中加入“从当前会话恢复配置”的逻辑。
4. 在 `iflowService.ts` 中将设置类 RPC 的失败改为可上抛或返回结构化结果。
5. 在 `chatView.ts` 中补失败提示。
6. 补最小测试护栏。

### 8.2 不推荐方案

以下方案本次不建议采用：

1. 直接重做整个顶部配置弹层。
2. 在本次功能中引入远程能力发现。
3. 将模型与模式配置完全迁移到全局设置页。
4. 重新定义一套新的配置存储结构。

原因：

1. 用户痛点核心不是 UI 不够复杂，而是状态没有闭环。
2. 当前数据结构已足够支撑本次修复。
3. 本次应优先解决真实生效与可恢复性，而非继续扩大范围。

## 9. TDD 开发顺序

### 9.1 第一阶段：会话设置同步

先补测试，再改实现。

测试目标：

1. 模型切换后，当前会话的 `model` 被更新。
2. 模式切换后，当前会话的 `mode` 被更新。
3. 思考切换后，当前会话的 `think` 被更新。

### 9.2 第二阶段：会话切换恢复

测试目标：

1. 切换到会话 A 后恢复 A 的 model/mode/think。
2. 切换到会话 B 后恢复 B 的 model/mode/think。
3. 顶部摘要与按钮状态同步刷新。

### 9.3 第三阶段：发送链路验证

测试目标：

1. 发送消息时使用恢复后的当前会话配置。
2. 新建会话后继承当前视图配置。

### 9.4 第四阶段：失败反馈

测试目标：

1. `session/set_model` 失败时用户有提示。
2. `session/set_mode` 失败时用户有提示。
3. `session/set_think` 失败时用户有提示。

## 12. 验收标准

满足以下条件方可视为完成：

1. 用户切换模型后，发送消息实际使用新模型。
2. 用户切换模式后，发送消息实际使用新模式。
3. 用户切换思考开关后，发送消息实际使用新思考状态。
4. 用户切换到已有会话时，配置控件恢复为该会话保存的值。
5. 用户新建会话时，会继承当前配置。
6. 若后端拒绝设置，用户能看到清晰提示。
7. 所有新增测试通过。
8. `npm run build` 通过。

## 13. 手工回归清单

### 13.1 单会话验证

1. 打开一个会话。
2. 切换模型。
3. 切换模式。
4. 开启思考。
5. 发送消息。
6. 观察响应与日志，确认请求链路没有异常。

### 13.2 跨会话验证

1. 会话 A 设为模型 A、模式 Plan、思考开启。
2. 会话 B 设为模型 B、模式 Smart、思考关闭。
3. 在 A 与 B 间反复切换。
4. 确认顶部摘要、按钮状态、发送行为均与各自会话一致。

### 13.3 新建会话验证

1. 当前会话选择某组配置。
2. 新建会话。
3. 确认新会话继承当前配置。

### 13.4 失败提示验证

1. 模拟后端不支持某个设置 RPC。
2. 发送消息。
3. 确认用户可见提示出现。

## 14. 风险与后续增强

### 14.1 本次已知风险

1. 若后端协议实际不支持某些设置 RPC，本次只能做到失败可见，不能强行伪造成功。
2. 若按钮 UI 刷新不完整，仍会让用户误判为功能不生效。
3. 若测试仅覆盖存储不覆盖发送链路，仍可能遗漏真实发送行为偏差。

### 14.2 后续增强建议

可作为下一阶段增量任务：

1. 动态从 iFlow CLI 拉取模型列表。
2. 动态从 iFlow CLI 拉取模式列表。
3. 在顶部摘要中加入“当前会话配置来源”的辅助信息。
4. 为设置类 RPC 增加统一的错误码映射与可读提示。
