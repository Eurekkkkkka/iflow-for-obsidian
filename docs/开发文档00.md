# iFlow 阶段优化开发文档

## 1. 文档目的

本文档用于承接基于用户体验与 UI 调研得出的阶段性优化计划，并将其落成可以直接执行的任务集合。目标不是一次性重写插件，而是在保持主流程稳定的前提下，分阶段解决以下问题：

1. 用户不知道当前发生了什么。
2. 用户不知道本轮会把哪些上下文发送给模型。
3. 消息内容、工具调用和思考内容混在一起，阅读成本高。
4. 顶部结构和高频交互效率不足。
5. 设置、诊断和错误恢复闭环不完整。

## 3. 阶段划分

### 阶段一：体验基线与验收标准

目标：固化用户旅程、失败分支和验收口径，避免后续优化目标漂移。

### 阶段二：运行时反馈与上下文可见性

目标：先解决“用户看不懂当前状态”和“用户不知道发送了什么”的问题。

### 阶段三：消息内容渲染与阅读质量

目标：提升聊天内容可读性，修复代码块、表格、列表和 Thinking 样式问题。

### 阶段四：思考流与工具流拆分

目标：将最终回答、模型思考、工具调用三类信息分开呈现，避免消息混流。

### 阶段五：顶部信息架构重组

目标：降低顶部常驻控件密度，提升当前任务聚焦度。

### 阶段六：高频交互效率优化

目标：减少重复点击、定位丢失和附件处理摩擦。

### 阶段七：设置页与诊断闭环

目标：把分散在日志和 Notice 中的恢复能力产品化。

### 阶段八：视觉精修与可访问性

目标：统一视觉语言并补齐深浅主题、窄侧栏和键盘可达性问题。

### 阶段九：测试、文档与发布准备

目标：为每个阶段补齐自动化测试、人工回归和发布说明。

## 4. 实施顺序

建议顺序如下：

1. 阶段二和阶段三并行。
2. 阶段四单独推进。
3. 阶段五和阶段六串行推进。
4. 阶段七、阶段八、阶段九作为收尾阶段。

## 5. 关键文件

1. `src/chatView.ts`：主视图装配层，是多数 UI 调整的核心落点。
2. `src/iflowService.ts`：连接、会话、模型、模式、思考开关设置入口。
3. `src/features/chat/toolCallMapper.ts`：思考流与工具流事件拆分的关键位置。
4. `src/ui/chat/messageRenderer.ts`：消息渲染与阅读质量的主要修改点。
5. `src/ui/chat/conversationSelectorView.ts`：会话列表、分组和切换体验的落点。
6. `src/ui/chat/imageAttachmentView.ts`：图片附件体验的基础模块。
7. `src/styles.css`：布局、状态样式、主题适配的主样式文件。
8. `src/main.ts`：启动流程、设置页、命令和诊断入口。

## 6. 验收原则

1. 每个阶段都必须能独立上线。
2. 每个阶段都必须能独立验收。
3. 涉及状态流、事件流、渲染规则的阶段必须补测试。
4. 所有阶段都要在深色主题、浅色主题和窄侧栏下验证。
5. 阶段四之后必须重点检查思考流不会再与最终回答混流。

## 7. 任务文档入口

每个阶段的每一个 task 都已经拆分为单独的 Markdown 文档，位于 `docs/stage-development-docs/` 下，命名规则为：

- `phase-x/task-x-y-*.md`

建议使用方式：

1. 先阅读当前阶段总目标。
2. 再按 task 文档逐项执行。
3. 每完成一个 task，就更新测试结果和风险说明。
4. 每完成一个阶段，就回写 `docs/stage-development-docs/dev-log.md` 。
